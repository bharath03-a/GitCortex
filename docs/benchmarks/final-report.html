Does a code knowledge graph help AI work more efficiently?

Measured Claude token usage on real developer questions, once with ordinary file search, once with the GitCortex graph (compact MCP). This is the median of 3 runs with errored/rate-limited sessions excluded — the noise-resistant view. Every number is from the actual API response.

What shipped in v0.5.0

  • 7 new MCP tools (15 → 22): graph_stats, ast_search, type_hierarchy, find_importers, find_type_usages, module_dependencies, get_call_sites.
  • Configurable response token budget (GCX_RESPONSE_BUDGET, default 2000): every list tool truncates to fit, so a high-fan-out symbol can never out-cost grep.
  • Edge confidence (extracted vs inferred): cross-file name-resolved edges are flagged lower-confidence — a precision signal.
  • Richer graph data: cyclomatic complexity, decorator/annotation metadata, and exact call-site lines.
  • Fixed: Rust import edges were silently dropped; search now handles space-separated and typo queries.
  • Tour: no-seed tours emit a component-level architecture summary with file:line, instead of a flat symbol list.
  • Trustworthy benchmark: harness now retries on rate limits, throttles, excludes errored sessions, and reports median-of-N — the prior single-run numbers were too noisy to trust.
◀ Previous published run — 2026-06-06

The 2026-06-06 run reported up to 56% savings (full MCP, large repos) and 1.15–1.36× compact geomean. Those were single-run numbers; this page supersedes them with median-of-N data and a rate-limit-resilient harness. Full archived report: final-report-2026-06-06.html.

Methodology

Model claude-haiku-4-5. Compact MCP (single dispatch tool). Per question, two Claude sessions (grep arm vs gcx arm), 3 rounds, sequential + throttled with retries. Tokens = input + cache-creation (cache reads excluded to avoid double counting). Errored sessions dropped. Reproduce: bash docs/benchmarks/stable-sweep.sh.
